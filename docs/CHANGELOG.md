- **해결 방법**: `/course/[id]` 공개 상세 페이지 신설, `/api/og`에서 코스 경로를 그린 OG 이미지를 동적 생성, `lib/kakaoShare.ts`의 `shareCourse()`로 카카오 SDK `Kakao.Share.sendDefault()` 호출.

### 2. OG 이미지가 카카오톡 카드에서 잘려 보임

- **문제 상황**: 카카오톡 공유 카드에서 경로 이미지가 잘려 보이거나 제목이 잘림.
- **원인**: (1) 캔버스가 1200x630(≈1.905:1)으로 카카오 Feed 템플릿이 기대하는 2:1과 미세하게 어긋나 크롭 발생. (2) 카카오 Feed 템플릿은 이미지를 **중앙 기준 정사각형**으로 추가 크롭하는데, 로고/제목이 좌상단·좌하단에 배치돼 있어 그 크롭 영역 밖으로 밀려남.
- **해결 방법**: 캔버스를 정확히 1200x600(2:1)로 변경. 경로 그림은 캔버스 전체 폭에 걸쳐 그리되(잘려도 배경 장식처럼 보이도록), 로고/제목/거리 텍스트는 중앙 600px 안전 영역(x: 300~900)에 오도록 재배치. `coursePreview.ts`의 `computeCoursePath`도 정사각형 전용에서 가로×세로 안전 영역을 받는 형태로 확장.
- **관련 커밋**: `0424178` (2:1 비율 전환), `012833d` (중앙 안전 영역 재배치)

### 3. OG 이미지가 카카오톡에서 계속 예전 버전으로 보임

- **문제 상황**: 코드는 수정했는데도 카카오톡 카드에는 여전히 예전(크롭된) 이미지가 노출됨.
- **원인**: `courseId`만으로 구성된 OG 이미지 URL이 고정돼 있어, 카카오/Vercel 엣지 캐시가 이전 결과를 계속 서빙함.
- **해결 방법**: `courses.created_at`을 버전 파라미터(`&v=`)로 붙여 코스마다 고유한 URL을 생성(`site.ts`의 `ogImageUrl`). `updated_at` 컬럼이 없어 `created_at`으로 대체.
- **관련 커밋**: `da16877`

### 4. 카카오톡 공유 카드에 "서비스 이용하기" 버튼 추가

- **문제 상황**: 공유 카드에 코스 링크로 가는 버튼만 있고, 서비스 자체로 유입시킬 버튼이 없었음.
- **해결 방법**: `sendDefault`의 `buttons` 배열(최대 2개)에 "코스 보기"에 이어 "서비스 이용하기"(→ `shaperun.kr`) 버튼 추가.
- **전제 조건(코드 변경 아님)**: 카카오 디벨로퍼스 콘솔의 "제품 링크 관리 → 웹 도메인"에 `shaperun.kr`이 등록되어 있어야 버튼이 정상 노출됨 — 사용자가 콘솔에서 등록 상태를 직접 확인함.
- **관련 커밋**: `f121dcf`

### 5. 지도 페이지에서 저장 직후 공유하면 이미지가 안 뜸 (CORS)

- **문제 상황**: my-courses에서 공유하면 이미지가 정상 노출되는데, 지도 페이지에서 저장 직후 공유하면 이미지 없이 텍스트만 뜸.
- **원인 조사**: `createdAt` 값 이상/undefined, 워밍업 타임아웃 등을 의심했으나 모두 정상이었음. 실제 원인은 두 가지였음.
  1. **레거시 중복 로직**: 지도 페이지 하단 액션바의 "공유" 버튼이 `shareCourse()`가 아니라 og-이미지/버전 파라미터가 전혀 반영 안 된 예전 인라인 구현(`handleKakaoShare`)을 쓰고 있었음. 저장 토스트(3초 후 소멸)의 "공유" 버튼과는 별개 경로였음.
  2. **CORS 차단**: `warmOgImageCache`가 절대경로(`https://shaperun.kr/...`, apex 도메인)로 fetch를 시도했는데, apex는 www로 308 리다이렉트되고 그 리다이렉트 응답에 `Access-Control-Allow-Origin` 헤더가 없어, www에서 접속 중인 사용자의 브라우저가 크로스 오리진 요청으로 판단해 차단함.
- **해결 방법**: (1) `handleKakaoShare` 제거, 하단바/토스트 버튼 모두 `shareCourse()` 하나로 통합. (2) `warmOgImageCache`의 fetch만 상대경로(`ogImagePath`)로 변경해 항상 현재 접속 중인 origin 기준으로 요청되게 함 (카카오 서버가 직접 fetch하는 `content.imageUrl`은 서버 간 요청이라 CORS 대상이 아니므로 절대경로 유지).
- **관련 커밋**: `efb237e`(진단 로깅), `b189563`(레거시 로직 통합), `6129477`(CORS 수정)

## 2026-07-09 — 코스 공유/공개 정책 정비

### 6. 코스 중복 다운로드 / 자기 코스 자기다운로드 방지

- **문제 상황**: 같은 코스를 여러 번 다운로드하거나, 본인이 만든 코스를 본인이 다운로드할 수 있었음.
- **해결 방법**: `downloadCourse()`에 본인 코스 체크(`OwnCourseDownloadError`) 및 unique constraint violation(23505) 핸들링(`AlreadyDownloadedError`) 추가. DB에는 `(user_id, original_course_id)` unique constraint 추가.
- **관련 커밋**: `45d205c` / **마이그레이션**: `004_courses_download_unique.sql`

### 7. 다운로드한 코스는 항상 비공개로, 공유는 원본 링크로

- **문제 상황**: 다운로드한 사본을 공개로 전환하면 원본과 같은 경로가 커뮤니티 피드에 중복으로 노출됨.
- **해결 방법**: my-courses에서 다운로드한 사본(`original_course_id` 있음)은 공개/비공개 토글 UI 자체를 숨김. 공유 버튼은 사본이 아니라 **원본**의 `/course/[id]` 링크를 공유하도록 변경. 원본이 삭제되었거나 원작자가 비공개로 전환한 경우 공유 버튼을 비활성 표시하고 이유를 안내(사본 자체 기능은 영향 없음). DB에는 `original_course_id is not null이면 is_public은 항상 false` CHECK 제약을 추가해 다른 경로로 공개 전환하는 것도 차단.
- **관련 커밋**: `ef52c9b` / **마이그레이션**: `005_downloaded_courses_always_private.sql`

### 8. 작성자 이름이 프로필 닉네임 변경에 실시간 반영 안 됨

- **문제 상황**: `courses.creator_name`/`original_user_name`이 코스 저장·다운로드 시점의 닉네임 스냅샷이라, 프로필에서 닉네임을 바꿔도 피드/상세 페이지에 반영되지 않음.
- **원인**: (1) `courses`와 `profiles` 사이에 PostgREST가 인식하는 FK 관계가 없어 `.select('*, profiles(nickname))')` 같은 embed join이 불가능(`PGRST200`). (2) `profiles` RLS가 "본인 행만 읽기"로 막혀 있어, 코드로 조회해도 다른 사람의 닉네임은 항상 빈 값이었음(anon으로 `count()` 해도 0). (3) `profiles` row 자체가 로그인 시 자동 생성되지 않고, 사용자가 닉네임을 직접 수정해야만 생성돼서, 기존 유저 15명 중 14명은 row가 아예 없었음.
- **해결 방법**:
  1. `courses.ts`에 `fetchNicknames(client, userIds)` 헬퍼 추가 — `profiles`를 별도 조회 후 코드에서 병합(embed join 대신). `creator_name`/`original_user_name` 컬럼과 저장 로직은 그대로 두고, 화면 표시 시점에만 `liveNickname || 스냅샷`으로 폴백.
  2. `profiles`에 공개 SELECT RLS 정책 추가.
  3. `auth.users` insert 시 `profiles` row를 자동 생성하는 `security definer` 트리거(`handle_new_user`) 추가 — 닉네임 우선순위: 카카오 메타데이터(`full_name`/`name`) → "러너"+랜덤.
  4. 기존 14명 백필: 카카오 메타데이터 → 가장 최근 `creator_name` → "러너"+랜덤 우선순위로 idempotent하게 insert.
- **반영 위치**: 커뮤니티 피드(`listPublicCourses`), `/course/[id]` 상세 페이지, my-courses 다운로드 코스의 "출처: OOO" 표시.
- **관련 커밋**: `279836b`, `2b3d312` / **마이그레이션**: `006_profiles_public_read.sql`, `007_profiles_auto_create_trigger.sql`, `008_profiles_backfill_existing_users.sql` (적용 순서: 007 → 008 → 006)

### 9. 비로그인 사용자의 저장/공유 버튼 클릭 시 안내 없음

- **문제 상황**: 로그인 안 한 상태에서 "저장"/"공유" 버튼을 누르면(저장 버튼은 아예 숨겨져 있거나) 아무 안내 없이 동작이 무시됨.
- **해결 방법**: `requireAuth()`로 로그인 여부를 가장 먼저 확인하는 분기를 추가. 비로그인이면 기존 토스트와 통일된 스타일의 팝업("로그인 후 사용 가능해요!" + 로그인하기 버튼)을 띄우고 Supabase insert/카카오 공유 호출까지 가지 않도록 차단. 로그인 O + 미저장 상태의 기존 "먼저 저장해주세요" 안내는 `requireAuth()` 통과 이후 그대로 유지(분기 우선순위: 로그인 → 저장 여부 → 정상 진행). "저장" 버튼은 기존에 비로그인 시 아예 숨겨져 있었는데, 이제 항상 보이고 클릭 시 로그인 팝업으로 안내.
- **관련 커밋**: `7d13c2e`

### 10. 로고 클릭 시 조회 중이던 코스가 하단바에 남음

- **문제 상황**: `/course/[id]`에서 "지도에서 보기"로 넘어온 상태에서 로고를 클릭(지도 위치 리셋)하면, 지도 위치는 리셋되는데 하단바(경로 정보)는 그대로 남음.
- **원인**: `/course/[id]`에서 넘어온 경로(`?wps=` 쿼리)와 사용자가 직접 그리는 경로가 동일한 `waypoints`/`segments` state로 관리되고 있었음. 로고 클릭 핸들러(`handleResetView`)는 지도 위치(GPS)만 리셋하고 이 state는 건드리지 않도록 되어 있어(과거에 "그리던 코스 보존" 목적으로 의도된 동작), 조회 목적으로 불러온 코스도 구분 없이 유지됨.
- **해결 방법**: `?wps=` 쿼리로 코스를 불러왔을 때만 `isViewingLoadedCourse` 플래그를 켜두고, 로고 클릭 시 이 플래그가 켜져 있으면 지도 위치 리셋과 함께 `resetTrigger`도 같이 발동시켜 경로/하단바까지 완전히 초기화. 사용자가 직접 그리던 경로는 이 플래그가 켜지지 않으므로 기존처럼 로고 클릭 후에도 유지됨.
- **관련 커밋**: `cc26cdb`
